Where to look
-------------
* **Exchange hashpartitioning(..., 200)** — This is the shuffle. In the Spark UI SQL graph it
  appears as **Exchange**; it is almost always a **stage boundary** (child stage produces
  shuffle write, parent stage consumes shuffle read).
* **HashAggregate(... partial_count)** — Map-side (partial) aggregation; reduces bytes sent
  across the shuffle when it applies.
* **HashAggregate(... count)** without "partial" — Reduce-side final aggregation after the shuffle.
* **PartitionFilters** — Shows partition pruning on the scan; empty or wrong predicates here
  often explain full-table scans on partitioned lakes.
* **ReadSchema** — Column pruning; extra columns here mean extra I/O and larger shuffle rows.

Spark UI signals (same query on real data)
------------------------------------------
* **SQL** tab: one Exchange between scan-side operators and the final aggregate.
* **Stages**: typically **two** stages for this pattern — stage 0 shuffle write, stage 1 shuffle read.
* **Stage 0** summary metrics: most input bytes; non-trivial shuffle write if groups are large.
* **Stage 1** summary metrics: shuffle read; watch for **skew** (max task >> median) on hot keys.
